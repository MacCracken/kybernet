A surprising number of these were **aarch64-silent**: the binary
cross-builds fine, the x86_64 harness passes, but the produced
aarch64 binary calls completely wrong syscalls. The pattern is
worth calling out as a class — anyone working on a future
audit should grep for raw `syscall(N, ...)` with a literal `N`
and treat each as a CRITICAL-aarch64 finding by default.

## CRITICAL-1 through CRITICAL-7 — raw syscall numbers vs aarch64

**Class:** silent arch portability failure
**Pattern:** code uses `syscall(N, ...)` with a literal integer
`N` (x86_64's number). On aarch64 the same number routes to a
completely different syscall.

| Site | x86_64 sysno | Means on x86_64 | Means on aarch64 |
|---|---:|---|---|
| `src/main.cyr:224` (TMP_SYMLINK) | 88 | `symlink` | `setfsgid` |
| `src/main.cyr:267` (emergency shell exec) | 59 | `execve` | `pipe2` |
| `src/main.cyr:270` (emergency shell fallback exec) | 59 | `execve` | `pipe2` |
| `src/main.cyr:271` (emergency shell child exit) | 60 | `exit` | (unallocated; EINVAL) |
| `src/main.cyr:762` (kybernet exit) | 60 | `exit` | (unallocated; EINVAL) |
| `src/lib/privdrop.cyr:58` (local `SYS_PRCTL = 157`) | 157 | `prctl` | `setpriority`+1 — shadowed stdlib's 167 |
| `src/lib/notify.cyr:10-12` (local `SYS_SOCKET=41`, `SYS_BIND=49`, `SYS_RECVFROM=45`) | 41/49/45 | socket family | `pipe2`/`setsockopt`/`getsockopt` |

**Why the gates didn't catch it.** The aarch64 cross-build *compiles*
clean — cyrius's cc5_aarch64 doesn't validate syscall numbers
against the target arch. Running the binary under TCG would surface
the kernel's `ENOSYS` / wrong-call behavior, but kybernet's harness
only runs on x86_64 KVM (sakshi's invariant-TSC requirement, transitive
via libro/patra). aarch64 boot validation needs real aarch64 hardware
or aarch64-emulated KVM — neither is on the standing CI runner.

**Fix.** Replace every raw `syscall(N, ...)` with the stdlib's
arch-dispatched wrapper (`sys_exit`, `sys_execve`, `sys_symlink`,
`sys_prctl`), OR `#ifdef CYRIUS_ARCH_*` the local enum so the
right number is picked per arch (`SYS_SOCKET/BIND/RECVFROM` in
notify.cyr, since no stdlib wrappers exist yet — upstream issue
filed at `cyrius/docs/development/issues/2026-05-11-kybernet-socket-syscall-wrappers.md`).

**Regression scope.** This finding subsumes the 1.0.0 `SYS_OPEN`
fix in `src/lib/sandbox.cyr` (caught in 1.1.0 work). Same root
cause — raw syscall numbers — but only one site was audited at
the time. 1.1.5 closes the rest of the surface; the standing rule
is now "no literal syscall numbers in kybernet source — go through
a wrapper or `#ifdef`-arch-dispatched enum."

**Closed by.** `src/main.cyr` (4 sites → `sys_symlink`, `sys_execve` ×2, `sys_exit`),
`src/main.cyr` final exit (`sys_exit(r)`), `src/lib/privdrop.cyr`
(removed shadowing `SYS_PRCTL = 157` — stdlib's per-arch
definition now wins), `src/lib/notify.cyr` (`#ifdef`-gated enum
with per-arch numbers).

## HIGH-1 — `status_buf[1]` stack overflow

**Site:** `src/lib/reaper.cyr:14`
**Pre-fix:** `var status_buf[1];`
**Issue.** `sys_waitpid(pid, &status_buf, WNOHANG)` writes a 4-byte
Linux `int wstatus` to the buffer. The 1-byte stack array means
the kernel writes 3 bytes past the array bound. Behavior depended
on cyrius's stack alignment (8-byte default makes the spill land
in padding, so this happened to be benign in practice), but the
property was accidental — a compiler rev that changes alignment
or layout would corrupt the next stack variable.

**Same class as 0.95.0's `signalfd_siginfo buf[16]→[128]`** fix
(a 16-byte stack buffer where the kernel writes 128 bytes per
read). The pattern: "kernel writes N bytes into our buffer, but
we declared the array smaller than N."

**Closed by.** `var status_buf[8];` — 4 bytes is the minimum,
8 keeps the natural i64 alignment that `load32(&status_buf)`
implicitly expects.

## HIGH-2 — `_mount_skipped[16]` BSS overflow

**Site:** `src/lib/mount.cyr:72`
**Pre-fix:** `var _mount_skipped[16];` with loop bounds-checked
at `_mount_skipped_count < 16`.
**Issue.** Same as HIGH-1 but at BSS rather than stack: the array
is 16 bytes (capacity 2 i64 ptrs), but the loop stores up to 16
i64 ptrs at offsets `0, 8, 16, ..., 120` — a 128-byte total span,
overflowing the array by 112 bytes. Practical impact bounded
because the harness/default config has 2 optional mounts
(`/dev/pts`, `/dev/shm`); if both happened to fail, BSS gets
clobbered.

**Same class as 0.95.0's `_mount_table[8]→[240]`** fix. The
mount module produced both regressions; the size-vs-count
constraint is non-obvious from the field declaration.

**Closed by.** `var _mount_skipped[128];` (16 slots × 8 bytes).

## HIGH-3 — PID-1 exit-path regression in event loop

**Site:** `src/main.cyr:741`
**Pre-fix:**

```cyrius
switch (tok) {
    case 1: ...
    case 2: handle_health_tick();
    case 3: handle_watchdog_tick();
    case 5: handle_notify_msg();
    default: return 0;       // ← kernel panic if reached
}
```

**Issue.** On any unexpected epoll event token, the event loop
returns 0 from `kybernet_run`, which returns to `main`, which
calls `sys_exit(0)`. PID 1 exiting cleanly is a kernel panic
("Attempted to kill init!"). Unexpected tokens can fire any time
the epoll registration grows out of sync with the dispatcher (e.g.,
adding a new fd source without updating the switch).

**Same class as 0.95.0's "main.cyr: PID 1 exit paths now call
do_shutdown() instead of returning"** finding. That fix closed
the main known exit paths; this case was missed at the time.

**Closed by.** Replace `default: return 0;` with a `klog` line
and a fall-through to the next iteration. The event loop continues;
no kernel panic; the operator sees the warning in dmesg/serial.

## MEDIUM-1 — `Str` vs `cstr` type-confusion across the logging surface

**Sites:**
- `src/main.cyr` — `handle_sigchld` (4 sites), `start_services` (3 sites including the cgroup path-builder inputs `create_service_cgroup(name)` / `move_to_cgroup(name, pid)`), `handle_health_tick`, `handle_watchdog_tick`
- 12+ `klog2 / slog` calls total

**Issue.** Argonaut returns service / health / watchdog names as
`Str` (boxed: header + cstr-ptr inside). kybernet's `klog2`, `slog`,
and `cgroup_*` helpers all expect cstr. Passing a Str where a
cstr is expected makes them read the Str's header bytes as
ASCII chars — output is garbage; cgroup path construction targets
random filesystem paths derived from the header layout.

**Why latent.** The default config (`argonaut_config_default()`)
populates an empty services vec. The 1.1.4 harness boots with
the default config, so none of the 12 sites fire. A production
config with N services would hit ALL of them every service start /
stop / health-fail / watchdog-kill.

**Why we caught one in 1.1.4 but missed the rest.** The 1.1.4
harness rewrite exercised `run_boot_stages`, which surfaced the
`klog2("boot: ", desc)` Str-as-cstr bug. The fix was point-only
(`str_data(desc)` at that one site). The pattern is repo-wide;
1.1.5 closes the rest.

**Closed by.** Wrap each `vec_get`-returned name in `str_data()`
once at top-of-loop, reuse the cstr ptr for all downstream
`klog2 / slog / cgroup_*` calls. The 1.1.3 cgroup path cache
keys on cstr pointer identity for its 1-slot LRU, so reusing
the same cstr ptr across the burst also preserves the LRU hit.

## LOW-1 — `_logbuf` no `plen` bounds check

**Sites:** `src/main.cyr` — `_log_write` (line 46), `klog2` (line 77).
**Issue.** Both functions compute `total = plen + mlen + 1` then
clamp `copylen = 254 - plen`, which underflows if `plen > 254` —
producing a huge unsigned `copylen` that `memcpy` would honor.
The `_logbuf[256]` array would get blown out.

**Why deferred (not closed in 1.1.5).** Every current caller passes
a short literal prefix (the longest is `"FATAL: required boot stage failed: "` at 35 chars). No reachable path
exercises the latent overflow. Adding a `plen` clamp is cheap
defensive code, but it dilutes the audit-doc signal — a 1.1.5
that ships "we found nothing exploitable but added 4 lines of
defensive bounds-checking anyway" is less useful as a precedent
than one that says "we found nothing exploitable, here's the
inline reasoning."

**Mitigation.** Inline comment block in `_log_write` and `klog2`
documenting the `plen ≤ 254` precondition + the current caller
audit table. Promote to MEDIUM (and fix) the moment a non-literal
prefix shows up.

## LOW-2 — `_mount_table[288]` at exact capacity

**Site:** `src/lib/mount.cyr` mount-table backing array.
**Issue.** 6 entries × 48 bytes per entry = 288 bytes exactly.
Adding a 7th mount entry (e.g., bpf, tracefs, debugfs as a third
optional mount) without growing the array overflows.

**Closed by.** Inline comment documenting the size↔entry-count
invariant + the next-bump target (`_mount_table[480]` for 10
entries, gives 4 entries of headroom). The 0.95.0 `[8]→[240]`
fix and this 1.1.5 finding both came from the same class
("the array is exactly the size needed, no slack"); commenting
the constraint makes the regression less likely.

## Adjacent — areas audited and clean

These categories were checked and produced no findings:

- **fd lifecycle / leaks.** Every `sys_open` pairs with a `sys_close`
  in-frame or assigns to a long-lived global (`g_log_fd`, `g_sfd`,
  `g_epfd`, `g_nfd`) closed at shutdown.
- **CLOEXEC discipline.** Closed in 1.1.2; re-verified clean for
  1.1.5 (no new `sys_open` sites since).
- **Signal handler context.** `handle_signal` and downstream
  handlers run from `epoll_wait` → `read_signal` → dispatch —
  never from an actual signal-handler frame, so async-signal-safety
  isn't a constraint. Reentrant SIGCHLD bursts are handled by the
  while-loop in `reap_zombies` draining all pending zombies before
  returning.
- **Cgroup path-cache correctness (1.1.3).** The pointer-identity
  assumption (1-slot LRU compares ptrs not contents) is safe under
  the current call shape — every cgroup-cache caller passes either
  a literal cstr (same ptr per process) or a `str_data(name_cs)`
  ptr from a single argonaut-side allocation (stable for the
  service's lifetime). The MEDIUM-1 fix above strengthens this
  guarantee by funneling everything through `str_data` once at
  top-of-burst.
- **Harness-mode abuse surface (1.1.4).** `kybernet.harness=1`
  triggers a clean shutdown after services start — worst case is a
  DoS via attacker-controlled cmdline, which is already root-only
  (`/proc/cmdline` is set by the bootloader). The substring-with-
  boundary-checks pattern was lifted verbatim from argonaut's
  1.6.x audit-hardened form and rejects every wrong-shape input
  argonaut's audit identified (`nokybernet.harness=1`,
  `kybernet.harness=11`).
- **Path injection in `cgroup_path` / `execute_tmpfiles`.** Service
  names come from argonaut, which validates them at config-load
  time. tmpfile paths come from `config_tmpfiles()`, also argonaut-
  validated. No user-controlled input reaches a syscall argument
  on the kybernet side.
- **Integer overflow / signed-unsigned.** `is_mounted`'s
  `n - tlen` had a 0.95.0 underflow fix; verified still in place
  (`if (tlen >= n) { return 0; }` guard at mount.cyr). No new
  sites surfaced.

## Standing rules added to the kybernet audit checklist

These should be checked first on every future kybernet audit and
referenced in every cyrius bump:

1. **No literal `syscall(N, ...)`** with `N` an integer. Either
   use a stdlib wrapper or define a `#ifdef CYRIUS_ARCH_*`-gated
   enum that names the number. The compiler accepts the literal
   form, the harness accepts it, the production aarch64 deployment
   does not.
2. **Stack/BSS arrays — units are bytes.** `var X[N]` is N bytes,
   not N slots. Sites that store N i64 pointers need `[N * 8]`.
   When in doubt, write the math inline in a comment next to the
   declaration.
3. **`Str` vs `cstr`** — Argonaut surface is mostly `Str`; kybernet
   logging + cgroup path helpers are cstr-only. Any
   `klog2 / slog / *cgroup_*` call site that takes a value derived
   from `vec_get` on an argonaut return needs `str_data()` first.
4. **PID-1 exit paths must call `do_shutdown()` or `klog`-and-continue.**
   Never `return 0` from `kybernet_run` directly. Kernel panic
   ("Attempted to kill init!") is the failure mode.
5. **Mount table size.** Update the backing array AND the per-entry
   stride comment together. The constraint is verified by the
   `test_mount_required_flag` regression in src/test.cyr — failing
   `assert_eq(load64(t + 5 * 48 + 40), 1, ...)` is the canary.
